# Q1. 자바스크립트에서 this 키워드의 사용과 그 특성에 대해 설명해 주세요.
<details>
  <summary>(1) 기본 개념</summary>

  * **실행 컨텍스트에 따라 그 값이 결정되며, 현재 실행중인 함수 또는 메서드의 "소유자"를 가리킴**
</details>

<details>
  <summary>(2) 상황별 this 값</summary>

  * **1. 일반 함수 호출**
  ```JavaScript
  function test() {
      console.log(this);
  }
  test();

  👉 브라우저: `window`
  👉 strict mode: `undefined`
  ```

  * **2. 객체의 메서드로 호출**
  ```JavaScript
  const obj = {
    name: '홍길동',
    greet() {
      console.log(this.name);
    }
  };
  obj.greet();

  👉 `this === obj`
  👉 점(.) 앞에 있는 객체가 this
  ```

  * **3. 생성자 함수 / 클래스**
  ```JavaScript
  function Person(name) {
      this.name = name;
  }
  const p = new Person("홍길동");

  👉 `this === 새로 생성된 객체`
  ```

  * **4. 화살표 함수⭐**

  ```JavaScript
  const obj = {
      name: "홍길동",
      greet: () => {
          console.log(this.name);
      }
  };

  👉 `this`는 자기 자신이 아니라, 바깥 스코프의 this를 그대로 사용

  // 결과 : undefined
  ```

  * **5. call / apply / bind**
  ```JavaScript
  function greet() {
      console.log(this.name);
  }

  greet.call({ name: "홍길동" });
  ```
</details>

<details>
  <summary>(3) 핵심 정리</summary>
  
  * **전역 실행 컨텍스트에서 this**: 전역 객체(브라우저에서는 window, Node.js에서는 global)를 가리킴
  * **일반 함수에서 this**: 기본적으로 전역 객체를 가리키지만, 엄격 보드('use strict')에서는 undefined가 됨
  * **객체의 메소드로서 함수가 호출**될 때, this는 해당 메소드를 호출한 객체에 바인딩 됨
  * **화살표 함수에서 this**: 함수가 생성된 시점의 실행 컨텍스트의 this를 **상속**받
</details>

<br>

# Q2. 렉시컬 스코프(Lexical Scope)의 개념과 그 특성에 대해 설명해 주세요.
<details>
  <summary>(1) 기본 개념</summary>

  * **렉시컬 스코프란?**
    * 변수와 블록 스코프가 코드가 작성된 구조에 따라 결정되는 스코프의 한 형태
    * 즉, 함수나 블록이 정의된 시점의 외부 환경을 기준으로 스코프가 결정됨
    * 이는 함수가 실행될 때가 아니라 선언될 때 스코프가 결정된다는 것을 의미
  * 예
    * 힘수 내부에서 다른 함수를 정의하고 호출하면, 내부 함수는 항상 자신의 정의된 환경을 참조함
    * 이러한 특성 때문에 클로저가 가능해지며, 외부 함수의 변수를 내부 함수에서 접근할 수 있음
    * * 클로저란, 함수와 그 함수가 선언될 당시의 렉시컬 환경의 조합을 말하는 것으로, 외부 함수의 변수를 내부 함수에서 접근할 수 있도록 해줌
</details>

<details>
  <summary>(2) 렉시컬 스코프의 작동 방식</summary>

  * **스코프 결정 시점**: 함수가 정의된 위치를 기준으로 스코프가 결정됨
  * **스코프 체인**: 함수 내부에서 변수를 참조할 때, 먼저 내부 스코프에서 변수를 찾고, 없으면 외부 스코프로 이동하여 변수를 찾는 과정
  * **상위 스코프 참조**: 내부 함수는 자신이 정의된 외부 환경(상위 스코프)을 참조할 수 있음. 이를 통해 함수가 호출되는 위치와 상관없이 일관된 스코프를 유지함
</details>

<details>
  <summary>(3) 호출 위치는 중요하지 않음</summary>

  ```JavaScript
  const x = 1;

  function outer() {
      const x = 2;

      function inner() {
          console.log(x);
      }

      return inner;
  }

  const fn = outer();
  fn();

  // 결과 : 2
  // 이유
  // - inner는 이미 outer의 스코프를 기억하고 있음
  // - 이게 바로 클로저(Closure)
  ```
</details>

<details>
  <summary>(4) 특징 정리</summary>

  * 스코프는 **정의 시점에 결정됨**
  * 실행 위치와 상관 없음
  * 내부 함수는 외부 함수의 변수 접근 가능
  * 클로저를 만들 수 있음
</details>

<details>
  <summary>(5) this vs 렉시컬 스코프(핵심 비교)</summary>

  |구분|this|렉시컬 스코프|
  |-----|-----|-----|
  |기준|호출 방식|선언 위치|
  |결정 시점|실행 시|정의 시|
  |변경 가능 여부|가능(call, bind 등)|불가능|
  |화살표 함수|상위 this 사용|영향 없음|
</details>

# Q3. Promise의 3가지 상태에 대해 설명하시오.

<details>
  <summary>(1) Promise란?</summary>

 * **주로 서버에서 받아온 데이터를 화면에 표시할 때 사용**
 * 데이터를 다 받아오기 전에 화면에 데이터를 표시하려고 하면 빈 화면이나 오류가 발생하는데, 이를 해결하기 위한 방법 중 하나가 바로 Promise
</details>

<details>
  <summary>(2) Promise의 3가지 상태(State)</summary>

  * **1. Pending(대기)**
    * 비동기 처리 로직이 완료되지 않은 초기 상태
    * 새로 생성될 때 기본적으로 이 상태에서 시작하며, 비동기 작업이 진행 중이라는 것을 의미함
    * 이 시점에서 프로미스는 성공(fulfilled) 또는 실패(rejected)로 상태가 변경될 수 있음
   
  * **2. Fulfilled(이행)**
    * 콜백함수 인자 resolve를 호출한 상태로, 비동기 작업이 성공적으로 완료되어 프로미스가 결과 값을 가지게 된 상태
    * fulfilled 상태가 되면, 프로미스에 등록된 then() 메서드의 첫 번째 콜백 함수가 호출되며, 이 함수는 비동기 작업의 결과를 처리함
   
  * **3. Rejected(실패)**
    * 비동기 처리가 실패하거나 에러가 발생하여 프로미스를 완료할 수 없는 상태
    * 프로미스가 예상된 결과를 제공할 수 없음을 나타내며, 이 경우 then() 메서드의 두 번째 콜백 함수 또는 catch() 메서드에 등록된 콜백 함수가 호출됨
    * 이 함수들은 예러 처리 또는 예외 처리를 담당하여 프로그램이 적절히 반응할 수 있도록 함
</details>

<br>

# Q4. 이벤트버블링과 캡쳐링을 설명하고, 이를 방지하기 위한 방법을 서술하시오.

* **이벤트 버블링(Event Bubbling)** 과 **캡처링(Event Capturing)** 은 **DOM 이벤트가 전파되는 방식**을 설명하는 개념으로, 핵심은 "이벤트가 한 요소에서 발생했을 떄, 부모 요소들과 어떤 순서로 상호작용하느냐임

<details>
  <summary>(1) 이벤트 버블링(Event Bubbling)</summary>

  * 이벤트가 **가장 하위 요소(taeget)** 에서 시작해서 **상위요소** 로 이벤트를 전파하는 방식
  * HTML 구조상 자식요소에서 발생한 이벤트가 상위의 부모요소에까지 영향을 미치는 것
  * 이벤트 핸들러가 중첩된 요소들에서 이벤트가 발생했을 때, 하위요소에서 상위요소로 전파되는 것
  * 흐름은 자식 → 부모 → 조상 → document/window 흐름으로 이어짐
  * 이벤트 위임 시 사용하는 게 좋음
  * **문제점**
    * 브라우저에서 예상치 못한 동작이 이뤄지는 경우가 빈번하기에, 미리 예방하거나 적절하게 이벤트를 차단해야함
</details>

<details>
  <summary>(2) 이벤트 캡처링(Event Capturing)</summary>

  * 이벤트 버블링과는 반대로 **상위요소에서 하위요소로의 이벤트 전파방식**을 의미함
  * addEventListener의 세번째 인자 값의 default가 버블링이기 때문에, 세 번째 인자 값에 true를 적용하면 이벤트 캡처링이 발생함
  * 특정 이벤트를 **먼저 가로채야 할 때**나 보안, 로깅, 글로벌 이벤트 처리 시 사용하면 좋음
</details>

<details>
  <summary>(3) 방지 방법</summary>

  * **event.stopPropagation()** 메서드 : 이벤트 전파 자체를 막음
  * **event.stopImmediatePropagation()** 메서드 : 같은 요소에 등록된 **다른 이벤트 리스너까지 모두 막음**
</details>

# Q5. 브라우저가 어떻게 동작하는지 설명하시오.
* **동작 원리**
  * 사용자가 URL을 입력하면 DNS를 통해 IP를 찾아 서버에 데이터를 요청(HTTP)하고, 받은 HTTP/CSS/JS 파일을 파싱하여 렌더링 엔진이 화면에 시각적으로 그려내는 과정(요청 → 응답 → 파싱 → 렌더링)임
  * 핵심은 렌더링 엔진이 DOM과 CSSOM을 결합해 렌더 트리를 생성한 후, 레이아웃과 페인트 과정을 거쳐 화면에 표시하는 것
 
<details>
  <summary>(1) 브라우저의 핵심 구조</summary>

  * **사용자 인터페이스(UI)** : 주소 표시줄, 이전/다음 버튼, 북마크 메뉴 등
  * **브라우저 엔진(Browser Engine)**: UI와 렌더링 엔진 사이의 동작을 제어
  * **렌더링 엔진(Rendering Engine)**: HTML, CSS를 파싱하여 화면에 표시
  * **통신(Networking)**: HTTP/HTTPS 요청 처리
  * **자바스크립트 해석기(JS Interpreter)**: JS 코드를 해석하고 실행
</details>

<details>
  <summary>(2) 브라우저 동작 순서(렌더링 과정)</summary>

  * **1. 입력 및 DNS 조회**: URL 입력 시 DNS 서버에서 IP 주소 조회
  * **2. HTTP 요청/응답**: 브라우저가 서버에 HTML 요청 후 수신
  * **3. 파싱(Parsing)**
    * **DOM 트리 생성**: HTML을 파싱하여 노드 트리(DOM) 구축
    * **CSSOM 트리 생성**: CSS를 파싱하여 스타일 구조(CSSOM) 구축
  * **4. JS 실행**: DOM 조작 
  * **5. 렌더 트리(Render Tree) 구축**: DOM과 CSSOM을 결합하여 화면에 표시될 요소만 선정
  * **6. 레이아웃(Layout/Reflow)**: 뷰포트 내에서 각 노드의 정확한 위치와 크기 계산
  * **7. 페인트(Paint)**: 계산된 요소를 실제 화면에 픽셀로 표현
  * **8. 최종 출력(Compositing)**
</details>

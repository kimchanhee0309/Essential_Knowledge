# Q1. Git에서 branch merge 방법과 각 방법의 특징
<details>
  <summary>(1) Create a merge commit</summary>

  * **1. 특징** 
    * 각 branch의 변경 사항(history)을 **모두 유지**하며 main branch에 다른 branch를 병합(merge)함
    * 각 branch의 변경 사항들이 과거의 커밋으로 보존됨
  * **2. 장점**
    * 프로젝트의 진행 상황을 **명확히 추적할 수 있음**
    * branch 별 **변경 사항이 유지**되므로 커밋들의 아이디가 바뀌지 않음
  * **3. 단점**
    * 병합이 될 때마다 커밋이 생기므로 다양한 branch를 병합하면 **커밋 로그가 매우 복잡해질 수 있음**
    * 대규모 인원이 참여하는 프로젝트에서는 **복잡도가 빠르게 증가**
</details>

<details>
  <summary>(2) Squash and merge</summary>

  * **1. 특징**
    * 병합할 branch의 모든 변경 사항을 **하나의 커밋으로 합쳐** main branch에 병합(merge)함
    * 병합되는 branch의 변경 사항들은 사라지며, 모든 변경 사항이 합쳐진 한 개의 커밋이 **새로운 아이디**를 가지고 main branch에 추가되어 병합(merge)됨
  * **2. 장점**
    * 커밋 히스토리를 **간단하게 유지**할 수 있음
    * 각 커밋들이 합쳐지며 **가장 중요한 내용이 담긴 하나의 커밋으로 압축됨**
  * **3. 단점**
    * 각 커밋에 담겨 있던 **세부적인 작업 이력을 잃게 됨**
    * 개별적인 맥락을 알 수 없으므로 **문제가 발생한 부분을 찾기 어려움**
    * 기존에 존재하던 커밋 아이디들이 하나의 새로운 아이디로 생성되므로 **기존 branch를 기반으로 한 작업에 문제가 발생할 수 있음**</br></br>
    ※ squash 방식을 사용한다고 해서 기존의 커밋 내용이 완전히 삭제되는 것은 아님
        → Git이 아닌 GitHub에는 기존의 커밋 내역들이 모두 남아있음
</details>

<details>
  <summary>(3) Rebase and merge</summary>

  * **1. 특징**
    * 현재 위치한 branch를 병합시킬 **target branch에 재위치(rebase) 시킨 뒤 병합**함
    * 기준이 되는 base branch를 변경(re-base)한다고 생각하면 됨
    * 현재 위치한 branch의 **모든 커밋을 target branch**로 옮기기 때문에 **커밋 아이디가 변경됨**
  * **2. 장점**
    * 커밋 히스토리가 선형적으로 깔끔하게 유지됨
    * 깨끗한 히스토리 덕분에 변경 사항들을 쉽게 파악할 수 있음
  * **3. 단점**
    * 기존에 존재하던 커밋 아이디들이 각각 새로운 아이디로 생성되므로, **기존 branch를 기반으로 한 작업에 문제가 생길 수 잇음**
    * 다양한 branch가 존재할 경우 병합이 어려움
    * 선형적이라는 특징으로 인해 **특정 기능이 어디부터 어디까지의 커밋으로 구현되었는지 알 수 없음**
</details>

<details>
  <summary>(4) Fast-forward merge</summary>

  * **1. 특징**
    * 병합 시 **새로운 merge commit을 생성하지 않고**, 현재 위치한 branch의 **최신 커밋(=HEAD가 가리키고 있는 커밋)** 이 병합할 branch의 **최신 커밋으로 이동**하며 병합됨
    * main branch에서 다른 branch가 분기되었지만, 다른 branch에서만 새로운 커밋이 생기고 main branch에서는 아무런 변화가 없는 경우 발생함
        
  * **2. 위 병합 방식과의 차이점**
    * squash and merge는 **다른 branch의 커밋들을 합쳐서 하나의 커밋으로 만든 뒤 병합**하지만, fast-forward merge는 **기존의 커밋들이 유지됨**
    * rebase and merge는 **main branch와 분기된 branch 둘 다에 새로운 커밋이 존재하는 상태에서 병합**이 진행되지만, fast-forward merge는 **main branch에는 변경 사항이 없는 상태에서 병합됨**
</details>

# Q2. Git Flow 브랜치 전략
<details>
  <summary>(1). branch 종류</summary>

</br>

  |branch 명|내용| 
|------|---|
|**main(master)**|실제로 제품이 출시되는 주 branch|
|**develop**|다음에 출시할 버전을 개발하는 주 branch|
|**feature**|추가적인 기능을 개발하는 보조 branch</br>develop branch에 포함됨|
|**release**|이번에 출시할 버전을 준비하는 보조 branch</br>develop branch에서 개발한 내용을 해당 branch로 옮겨와 QA 및 테스트 진행|
|**hotfix**|출시된 버전(main branch)에서 발생한 버그를 수정하는 보조 branch|
</details>

<details>
  <summary>(2) 주 branch</summary>

  * **1. main branch**
    * 실제로 운영되고 있는 서비스
    * 항상 안정적으로 배포가 가능한 상태의 코드만을 관리함
    * So, 해당 branch에서 검증되지 않은 branch를 병합하는 경우 큰일이 일어날 수 있음
       
  * **2. develop branch**
    * 다음에 배포할 내용을 개발하는 branch
    * 평소에 해당 branch를 기반으로 개발을 진행하게 되며, 개발이 완료되면 main branch에 병합(merge)됨
</details>

<details>
  <summary>(3) 보조 branch</summary>

  * **1. feature branch**
    * develop branch를 기반으로 분기되어 **새로운 기능을 개발할 때 사용하는 branch**
    * 해당 branch에서 자유롭게 개발한 뒤, 개발이 완료되면 다시 develop branch에 병합(merge)함
    * 보통 개발자의 개인 저장소에만 존재하기 때문에 원격 저장소(origin)에는 push하지 않음
      
  * **2. release branch**
    * develop branch를 기반으로 분기되어 **이미 개발된 기능을 점검할 때 사용하는 branch**
    * 해당 branch에서 버그가 발견되면 이를 수정하고, develop branch에도 동일하게 적용시켜 주어야 함
    * 점검을 마친 후 배포 가능한 상태가 되면, main branch에 바로 병합(merge)하거나 develop branch에 병합(merge)함
      
  * **3. hotfix branch**
    * 배포가 완료된 버전에서 급하게 고쳐야 할 사항을 수정하기 위해 사용하는 branch
    * 배포가 완료된 버전에서의 수정 사항이므로 main branch에서 분기되어 나옴
    * 수정을 완료한 내용은 develop branch에도 동일하게 적용시켜 주어야 함
    * 수정을 마친 후 배포 가능한 상태가 되면, main branch에 바로 병합(merge)하거나 develop branch에 병합(merge)함
    * 보통 hotfix branch는 **긴급하게 버그를 수정할 때 사용**되는 branch이므로 과정이 완료되면 해당 branch를 삭제해줌
</details>

<details>
  <summary>(4) 장단점</summary>

  * 장점 : **역할 분담이 명확**하고, 배포 전후 **관리가 매우 꼼꼼함**
  * 단점 : **구조가 복잡**하고, branch 이동과 merge 과정이 번거로움
    
</details>

<details>
  <summary>(5) 예시</summary>

  * 1단계 : 새로운 기능을 개발하기 위해 개발자는 **기존의 develop branch에서 새로운 feature branch를 생성**하여 작업 시작
  * 2단계 : feature branch에서의 작업이 완료되면 **다시 기존의 develop branch로 병합(보통 pull request로 코드 리뷰를 받은 뒤에 병합)**
  * 3단계 : 다음에 출시할 버전을 준비하기 위해 **기존의 develop branch에서 새로운 release branch를 생성**하여 코드를 점검
  * 4단계 : 충분한 테스트를 거친 후, **main branch로 병합**하여 제품 출시
  * 5단계 : 배포한 제품에서 버그가 발견되었다면, hotfix branch를 생성하여 수정
</details>

# Q1. 리액트가 렌더링 하는 방식을 설명하세요

* **브라우저 렌더링과 리액트 렌더링**
  * **브라우저 렌더링** : HTML과 CSS 리소스를 기반으로 웹페이지에 필요한 UI를 그려내는 과정
  * **리액트 렌더링** : 브라우저가 렌더링에 필요한 DOM 트리를 만드는 과정
    * 리액트 애플리케이션 트리 안에 있는 모든 컴포넌트들이 현재 자신들이 가지고 있는 `props`와 `state`의 값을 기반으로 어떻게 UI를 구성하고 이를 바탕으로 어떤 DOM 결과를 브라우저에 제공할 것인지 계산하는 일련의 과정

* **리액트 렌더링이 일어나는 경우**
  * 최초 렌더링
  * 리렌더링
    * 클래스형 컴포넌트의 `setState`가 실행되는 경우
    * 클래스형 컴포넌트의 `forceUpdate`가 실행되는 경우
    * 함수형 컴포넌트의 `userState()`의 두 번째 배열요소인 setter가 실행되는 경우
    * 함수형 컴포넌트의 `useReducer()`의 두 번째 배열요소인 dispatch가 실행되는 경우
    * 컴포넌트의 `key props`가 변경되는 경우
    * `props`가 변경되는 경우
    * 부모 컴포넌트가 렌더링되는 경우
      
* **리액트 렌더링 과정**
  * 1. 렌더링 프로세스가 시작되면 리액트는 컴포넌트 루트부터 아래로 내려가며 업데이트가 필요하다고 지정돼있는 컴포넌트를 찾음
  * 2. 업데이트가 필요한 컴포넌트를 만나면
    * 클래스형 컴포넌트는 클래스 내부의 **render()** 함수를 실행
    * 함수형 컴포넌트는 **FunctionComponent()** 그 자체를 호출하고, 결과물을 저장
  * 3. 렌더링 결과물은 JSX 문법으로 구성되어 있고, 자바스크립트로 컴파일되면서 `React.createElement()`를 호출하는 구문으로 변환됨
    * 이때, `createElement()`는 자바스크립트 객체를 반환함
  * 4. 이 렌더링 결과물을 수집하여, 리액트의 새로운 트리인 가상 DOM과 비교해 실제 DOM에 반영하기 위한 모든 변경사항을 차례차례 수집함(**Reconciliation**)

  * 리액트 렌더 과정은 크게 **렌더단계**와 **커밋단계**로 나뉨
    * **렌더단계**
      * **컴포넌트를 렌더링하고 변경사항을 계산하는 모든 작업**
      * 렌더링 프로세스에서 컴포넌트를 실행하여 이 결과와 이전 가상 DOM을 비교하는 과정을 거쳐 변경이 필요한 컴포넌트를 체크하는 단계
      * `type`, `props`, `key` 중 하나라도 변경된 것이 있다면 변경이 필요한 컴포넌트로 체크됨
        
    * **커밋단계**
      * **렌더 단계의 변경사항을 실제 DOM에 적용해 사용자에게 보여주는 단계**
      * 이 단계가 끝나야 비로소 브라우저 렌더링이 발생함

  * 리액트가 먼저 DOM을 커밋 단계에서 업데이트 한다면 이렇게 만들어진 모든 DOM 노드 및 인스턴스를 가리키도록 리액트 내부의 참조를 업데이트함
  * 그 다음 생명주기 개념이 있는 클래스형 컴포넌트에서는 `componentDidMount`, `componentDidUpdate` 메서드를 호출하고, 함수형 컴포넌트에서는 `useLayoutEffect` 훅을 호출함
  * 중요한 사실은 **리액트의 렌더링이 일어난다고 해서 무조건 DOM 업데이트가 일어나는 것은 아님**
  * 렌더리를 수행하여 변경사항을 감지했으나 변경사항이 없다면, 커밋 단계는 생략될 수 있음
  * 즉, 리액트의 렌더링은 가시적인 변경이 일어나지 않아도 발생할 수 있음
    
* **리액트 동시성 렌더링(Concurrent Rendering)**
  * **렌더와 커밋 단계로 이뤄진 리액트 렌더링은 항상 동기적으로 작동함**
  * So, 렌더링 과정이 길어질수록 앱의 성능저하로 이어지고, 결과적으로 그 시간만큼 브라우저의 다른 작업을 지연시킬 가능성이 있음
    * why? 순서가 보장되지 않는 비동기 방식으로 이뤄질 경우, 사용자가 하나의 상태에 대해 여러 다른 UI를 보게될 수 있기 때문임
  * But, 때로는 비동기 렌더링이 유효할 수 있는데, 가령 B컴포넌트의 렌더링 작업이 무거워 상대적으로 빠르게 렌더링 할 수 있는 A 컴포넌트라도 변경해서 보여줄 수 있는 경우!
  * 이처럼 **의도된 우선순위로 컴포넌트를 렌더링해 최적화할 수 있는 비동기 렌더링, 이른바 동시성 렌더링이 리액트 18에서 도입됨**

# 02. 리액트에서 Virtual DOM이 무엇인지, 이를 사용하는 이유는 무엇인지 설명해 주세요.

<details>
  <summary>1. Virtual DOM이란?</summary>

  * 실제 브라우저의 DOM(Document Object Model)을 그대로 조작하는 대신, **메모리 안에 가볍게 복사된 DOM 구조를 만들어 관리하는 방식**
  * 일반적으로 웹 브라우저는 HTML을 파싱해서 실제 DOM 트리를 만들고, 이 DOM을 직접 수정하면 화면이 바뀜
  * But, 이 과정은 비용이 큼(특히 빈번한 업데이트 시)
  * So, React는 **재조정(Reconciliation) 과정**을 통해 해결하고자 함
    * 실제 DOM과 똑같은 구조를 **자바스크립트 객체 형태로 메모리에 생성**
    * 상태(state)가 바뀌면 새로운 Virtual DOM 생성
    * 이전 Virtual DOM과 비교(diff)
    * 바뀐 부분만 실제 DOM에 반영
</details>

<details>
  <summary>2. 동작 흐름</summary>

  * 1) 초기 렌더 -> Virtual DOM 생성 -> 실제 DOM 반영
  * 2) 상태 변경 발생
  * 3) 새로운 Virtual DOM 생성
  * 4) 이전 Virtual DOM과 비교(diff 알고리즘)
  * 5) 변경된 부분만 실제 DOM에 업데이트
</details>

<details>
  <summary>3. Virtual DOM을 사용하는 이유</summary>

  * **1) 성능 최적화**
    * 실제 DOM 조작은 느림
      * DOM 변경 -> 레이아웃 계산 -> 리페인트 발생
      * 이 과정이 반복되면 성능 저하
    * Virtual DOM은 **필요한 부분만 업데이트**해서 효율적임
    
  * **2) 선언형 UI(Declarative UI)**
    * React는 "어떻게 바꿀지"가 아닌, **"이 상태면 UI는 이렇게 생겨야 한다"** 를 정의함
    * Virtual DOM은 이를 가능하게 해줌
      
  * **3) 코드 단순화 & 유지보수성 향상**
    * 개발자는 **DOM을 직접 조작 안해도 되고**, **상태만 관리**하면 UI가 자동으로 갱신됨
    * 즉, 복잡한 UI 로직이 훨씬 단순해짐
      
  * **4) 배치 업데이트(Batching)**
    * 여러 상태 변경을 **한 번에 모아서 처리**하여 불필요한 렌더링을 감소시킴
</details>

<details>
  <summary>4. 오해하면 안 되는 점</summary>

  * Virtual DOM이 항상 빠른 건 아니며, 핵심은 **"불필요한 DOM 업데이트를 줄인다"** 임
  * 즉, 작은 앱에서는 차이가 거의 없으나, 큰 앱에서는 효과가 큼
</details>

<details>
  <summary>5. 정리</summary>

  * Virtual DOM은 **"실제 DOM을 직접 건드리지 않고, 변경 사항만 효율적으로 반영하기 위한 메모리 기반 중간 계층** 임
</details>

<br>

# 03. 리액트에서 컴포넌트란 무엇이며, 함수형 컴포넌트와 클래스 컴포넌트의 차이점을 설명하세요.

<details>
  <summary>1. 리액트에서 컴포넌트란?</summary>

  * UI를 구성하는 **독립적이고 재사용 가능한 최소 단위**
  * 화면을 버튼, 카드, 리스트처럼 **조각으로 나눈 각각의 블록**
  * 특징
    * **입력**: props(부모로부터 데이터 받음)
    * **내부 상태**: state(컴포넌트 내부 데이터)
    * **출력**: JSX(화면에 렌더링될 UI)
    * 즉, **입력을 받아 UI를 반환하는 함수(또는 클래스)**
</details>

<details>
  <summary>2. 함수형 컴포넌트 vs 클래스 컴포넌트</summary>

  * **(1) 함수형 컴포넌트(Functional Component)**
    ```JavaScript
    function Greeting(props) {
        return <h1>Hello, {props.name}</h1>;
    }

    or

    const Greeting = ({ name }) => <h1>Hello, {name}</h1>;
    ```
    * **특징**
      * 단순한 JavaScript 함수
      * `this` 없음
      * 코드가 짧고 직관적
      * 과거에는 state 사용 불가 -> 지금은 Hooks로 가능
    
  * **(2) 클래스 컴포넌트(Class Component)**
    ```JavaScript
    class Greeting extends React.Component {
      render() {
        return <h1>Hello, {this.props.name}</h1>;
      }
    }
    ```
    * **특징**
      * ES6 클래스 기반
      * `this` 사용
      * state, lifecycle 메서드 사용 가능
    
</details>

<details>
  <summary>3. 핵심 차이 정리</summary>

  |구분|함수형 컴포넌트|클래스 컴포넌트|
  |---|---|---|
  |문법|함수|클래스|
  |코드 길이|짧고 간결|비교적 길고 복잡|
  |state 사용|Hooks로 사용(`useState`)|`this.state`|
  |lifecycle|Hooks(`useEffect`)|lifecycle 메서드|
  |this 사용|없음|있음|
  |가독성|높음|상대적으로 낮음|
</details>

<details>
  <summary>4. 현대 리액트의 흐름</summary>

  * **함수형 컴포넌트 + Hooks가 표준**
  * why?
    * 코드가 더 간결함
    * 재사용성 높음(custom hooks)
    * 로직 분리 쉬움
    * 클래스의 `this` 문제 없음
</details>

<details>
  <summary>5. 정리</summary>

  * 컴포넌트는 **UI를 구성하는 재사용 가능한 블록**
  * 함수형 vs 클래스 차이 핵심은 **Hooks 기반 함수 vs this 기반 클래스**
</details>

# 04. useMemo, useCallback에 대해 설명하고, 어떤 경우에 사용하면 좋을지, 남용할 경우 발생할 수 있는 문제점을 설명해 주세요.

* `useMemo`와 `useCallback`은 React에서 **불필요한 연산이나 렌더링을 줄이기 위한 최적화 도구**

<details>
  <summary>1. useMemo</summary>

  * **값을 메모이제이션(캐싱)** 하는 훅
    ```JavaScript
    const memoizedValue = useMemo(() => {
      return expensiveCalculation(a, b);
    }, [a, b]);

    - [a, b]가 바뀌지 않으면: 이전에 계산한 값을 그대로 재사용
    - 바뀌면 함수를 다시 실행해서 값을 새로 계산
    ```

  * **언제 쓰면 좋을까?**
    * **1) 계산 비용이 큰 경우**
      ```JavaScript
      const sortedList = userMemo(() => {
        return items.sort((a, b) => a.price - b.price);
      }, [items]);

      // 정렬, 필터링, 복잡한 연산 같은 것에 쓰면 좋음
      ```
      
    * **2) 참조 동일성이 중요한 경우**
      ```JavaScript
      const options = useMemo(() => ({ theme: "dark" }), []);

      // 객체/배열을 매번 새로 만들면 자식 컴포넌트가 계속 리렌더링됨
      ```
</details>

<details>
  <summary>2. useCallback</summary>

  * **함수를 메모이제이션(캐싱)** 하는 훅
    ```JavaScript
    const handleClick = useCallback(() => {
      console.log(count);
    }, [count]);
    ```

  * **useMemo와의 차이**
    * `useMemo` -> 값 캐싱
    * `useCallback` -> 함수 캐싱
    * 사실 내부적으로는 거의 동일함
   
  * **언제 쓰면 좋을까?**
    * **1) 자식 컴포넌트에 함수 props를 줄 때**
      ```JavaScript
      const Child = React.memo(({ onClick }) => {
        console.log("자식 렌더링");
        return <button onClick={onClick}>클릭</button>;
      });
      ```
      ```JavaSCript
      const handleClick = useCallback(() => {
        console.log("클릭");
      }, []);

      // 이걸 안 쓰면
      // - 부모 렌더링될 때마다 함수가 새로 생성됨
      // - React.memo 무효화됨
      // - 자식도 계속 리렌더링
      ```
      
    * **2) 의존성 배열 안정화**
      ```JavaScript
      useEffect(() => {
        doSomething();
      }, [handleClick]);

      // 함수가 계속 바뀌면 useEffect도 계속 실행됨
      // useCallback으로 고정 필요
      ```
    
</details>

<details>
  <summary>3. 핵심 차이 정리</summary>

  |구분|useMemo|useCallback|
  |---|---|---|
  |목적|값 캐싱|함수 캐싱|
  |반환값|계산 결과|함수|
  |사용 이유|무거운 연산 최적화|불필요한 리렌더 방지|
  
</details>

<details>
  <summary>4. 남용하면 생기는 문제</summary>

  * **1) 오히려 성능이 더 나빠짐**
    * `useMemo`, `useCallback`도 비용이 있음
      * 의존성 비교 비용
      * 메모리 유지 비용
      * 간단한 연산에 쓰면 손해
      ```JavaScript
      const value = useMemo(() => count + 1, [count]); // 의미 없음
      ```
       
  * **2) 코드 가독성 저하**
    ```JavaScript
    const fn = useCallback(() => {...}, [a, b, c, d]);
    ```
    * 의존성 배열이 많아지면 읽기 어렵고, 유지보수가 힘듦
    
  * **3) 잘못된 의존성 -> 버그**
    ```JavaScript
    const fn = useCallback(() => {
      console.log(count);
    }, []);  // count 빠짐

    // 최신 값이 아니라 옛날 값(클로저 문제) 사용됨
    ```
    
  * **4) 무조건 쓰면 좋다 착각**
</details>

<details>
  <summary>5. 실무 기준(중요)</summary>

  * **써야 하는 경우**
    * 무거운 계산(정렬, 필터, 변환)
    * `React.memo`와 같이 쓸 때
    * 함수 참조 안정성이 필요한 경우
    * 의존성 때문에 useEffect가 불필요하게 실행될 때
      
  * **안 써도 되는 경우**
    * 단순 계산(`count + 1`)
    * 렌더링 비용이 작은 컴포넌트
    * 자식이 memoized 되어 있지 않을 때
</details>

<details>
  <summary>6. 정리</summary>

  * `useMemo`, `useCallback`은 **성능 문제 있을 때만 쓰는 도구**이지 기본 옵션이 아님
</details>

# 05. 리액트 생명주기(life cycle)에 대해 설명해 주세요.

<details>
  <summary>1.전체 흐름(큰 그림)</summary>

  * **React 생명주기 3단계**
    * **1) Mount(처음 생성)**
    * **2) Update(상태/props 변경)**
    * **3) Unmount(제거)**
</details>

<details>
  <summary>2. Mount(마운트: 처음 렌더링)</summary>

  * **컴포넌트가 처음 화면에 나타나는 순간**
  * **흐름**
    * **1) 함수 실행(렌더링)**
    * **2) DOM 생성**
    * **3) 화면에 반영**
    * **4) side effect 실행**
    ```JavaScript
    useEffect(() => {
      console.log("마운트됨");
    }, []);

    // [] 의존성 배열이면 딱 한 번 실행(마운트 시)
    ``` 
</details>

<details>
  <summary>3. Update(업데이트)</summary>

  * **state 또는 props가 바뀔 때 발생**
  * **흐름**
    * **1) 함수 다시 실행(렌더링)**
    * **2) Virtual DOM 비교**
    * **3) 바뀐 부분만 실제 DOM 업데이트**
    * **4) effect 실행**
      ```JavaScript
      useEffect(() => {
        console.log("업데이트됨");
      }, [count]);

      // count가 바뀔 때마다 실행
      ```
</details>

<details>
  <summary>4. Unmount(언마운트)</summary>

  * **컴포넌트가 화면에서 사라질 때**
  * **cleanup 함수 실행됨**
    ```JavaScript
    useEffect(() => {
      return () => {
        console.log("언마운트됨");
      };
    }, []);

    // 주로 이벤트 제거, 타이머 정리, 구독 해제
    ```
</details>

<details>
  <summary>5. 핵심: useEffect 하나로 다 표현됨</summary>

  * 과거(class 컴포넌트)
    * `componentDidMount`
    * `componentDidUpdate`
    * `componentWillUnmount`
  * 지금은 전부 `useEffect`로 처리
</details>

<details>
  <summary>6. 대응 관계</summary>

  |개념|Hooks|
  |---|---|
  |Mount|useEffect(() => {}, [])|
  |Update|useEffect(() => {}, [deeps])|
  |Unmount|useEffect cleanup|
</details>

<details>
  <summary>7. 렌더링 vs effect</summary>

  * **렌더링**
    ```JavaScript
    function App() {
      console.log("렌더링");
      return <div>Hello</div>;
    }

    // 상태 바뀌면 무조건 실행됨
    // UI 계산 단계
    ```

  * **effect**
    ```JavaScript
    useEffect(() => {
      console.log("effect 실행");
    });

    // 렌더링 끝난 후 실행
    // "부수 효과" 처리
    ```
</details>

<details>
  <summary>8. 실행 순서 정리</summary>

  * **처음 렌더링(Mount)** : 렌더링 -> DOM 반영 -> useEffect 실행
  * **업데이트** : 렌더링 -> DOM 업데이트 -> 이전 effect cleanup -> 새 effect 실행
  * **언마운트** : cleanup 실행
</details>

<details>
  <summary>9. 중요한 특징들</summary>

  * **1) cleanup은 다음 effect 전에 실행됨**
    ```JavaScript
    useEffect(() => {
      console.log("effect");

      return () => {
        console.log("cleanup");
      };
    }, [count]);

    // 실행 순서: cleanup -> effect
    ```

  * **2) Strict Mode에서 두 번 실행(개발 환경)**
    * 개발 모드에서 일부 effect가 두번 실행됨
    * why? 사이드 이펙트 버그 잡기 위해
   
  * **3) 렌더링은 생각보다 자주 일어남**
    * So, effect를 잘 관리해야 함
</details>

<details>
  <summary>10. 한 눈에 정리</summary>
  
  ```
  [Mount]
  렌더 -> effect

  [Update]
  렌더 -> cleanup -> effect

  [Unmount]
  cleanup
  ```
  * React 생명주기는 **렌더링 -> DOM 반영 -> effect 실행 -> cleanup의 반복 구조**

</details>

# 06. 리액트에서 배열을 렌더링할 때 key를 설정해야 하는 이유와 key 설정 시 주의할 점을 설명하세요.

<details>
  <summary>1. key를 설정해야 하는 이유</summary>
  
  * Virtual DOM의 **재조정(Reconciliation) 과정에서 고유한 식별자를 제공하여 성능을 최적화하고, 컴포넌트의 상태를 올바르게 유지하기 위해서**
  * `key`가 없으면 리액트는 배열의 요소가 추가, 삭제, 변경될 때마다 모든 요소를 다시 렌더링하여 성능 저하가 발생할 수 있음
  * **주요 이유와 동작 원리**
    * **1) 효율적인 DOM 업데이트(성능 향상)**
      * 리액트는 기존 Virtual DOM과 새로운 Virtual DOM을 비교(Diffing)하는데, `key`값을 기반으로 이전의 요소와 현재 요소를 매핑하여 변경된 부분만 실제 DOM에 반영
    * **2) 컴포넌트 상태 유지**
      * 리스트의 순서가 변경되거나 항목이 삽입/삭제될 때, `key`는 컴포넌트의 상태(state)나 폼 입력 값 등을 고유한 식별자에 맞게 올바르게 매칭되도록 보장함
    * **3) 고유성 보장**
      * `key`는 형제 요소 간에 고유해야 하며, 일반적으로 데이터의 고유 ID를 사용하는 것이 좋음
    ```JavaScript
    // 좋은 예시: 고유한 ID 사용
    {todos.map(todo => (
      <ListItem key={todo.id} text={todo.text} />
    ))}
    ```
</details>

<details>
  <summary>2. key 설정 시 주의사항</summary>

  * 배열의 인덱스를 `key`로 사용하는 것은 리스트가 정렬되거나 필터링될 때 상태가 꼬일 수 있어 권장하지 않음
  * **1) 고유한 key값 사용**
    * Key 값이 중복되면, 어떤 요소를 변경했는지 판단할 수 없어, 의도치 않은 동작이 발생할 수 있음
      
  * **2) 인덱스 값 Key 사용 금지**
    * 인덱스 값은 변할 수 있는 값이므로, 1번 규칙에 위배 되는 결과를 초래함
    * 단, 리스트나 배열의 변동이 없는 경우, 인덱스를 Key로 사용할 수 있음
      ```JavaScript
      // 예외 상황: 리스트가 정적일 때 인덱스를 Key로 사용
      const staticItems = ['Item1', 'Item2', 'Item3'];
      const listItems =  staticItems.map((item, index) => <li key=(index)>{item}</li>);
      ```
      
  * **3) 안정적인 고유 값 사용**
    * Key는 고유 식별자를 사용하는 것이 좋음
</details>

# 07. 웹 페이지 렌더링 방식 CSR, SSR, SSG 각각의 특징과 각 방식을 어떤 상황에 사용하면 좋을지 설명해 주세요.

<details>
  <summary>1. CSR(Client-Side Rendering)</summary>

  * **브라우저(클라이언트)에서 렌더링**
  * **동작 방식**
    * 서버는 거의 빈 HTML + JS 파일 전달
    * 브라우저가 JS 다운로드
    * JS 실행 -> 화면 생성
   
  * **특징**
    * 초기 HTML은 거의 비어 있음
    * 이후 모든 렌더링은 브라우저에서 수행
    * 즉, 최초에 빈 HTML을 받고, 자바스크립트가 브라우저에서 전체 페이지를 렌더링함
   
  * **장점**
    * 페이지 전환이 빠름(SPA 느낌)
    * 서버 부담 적음
    * 사용자 인터랙션이 많은 앱에 유리(UX가 좋음)
   
  * **단점**
    * 초기 로딩 느림(JS 다운로드 + 실행 필요)
    * SEO(검색 엔진 최적화) 불리
    * 저사양 기기에서 성능 저하
   
  * **언제 쓰면 좋은가?**
    * 사용자 중심 인터랙션이 많은 서비스: 관리자 페이지(Admin), 대시보드, 로그인 이후 서비스 등
    * 즉, 상호작용이 많고 SEO가 중요하지 않은 경우
    * 예: React SPA 구조, Vue.js 앱   
</details>

<details>
  <summary>2. SSR(Server-Side Rendering</summary>

  * **서버에서 HTML을 만들어서 내려줌**
  * **동작 방식**
    * 요청이 오면 서버에서 HTML 생성
    * 완성된 HTML 전달
    * 브라우저는 바로 화면 표시
    * 이후 JS가 붙으면서 인터랙션 가능(Hydration)
   
  * **특징**
    * 매 요청마다 HTML 생성
    * 초기 화면이 완성된 상태로 옴
    * 즉, 사용자가 요청할 때마다 서버에서 데이터를 가져와 HTML을 완성하여 전송함
   
  * **장점**
    * 초기 로딩 빠름(첫 화면 바로 보임)
    * SEO에 유리
    * 소셜 공유(메타태그) 유리
  * **단점**
    * 매번 서버에 요청하여 화면을 생성하므로 서버 부하 증가
    * 응답 속도 서버 의존
    * 구현 복잡
  * **언제 쓰면 좋을까?**
    * SEO가 중요하고 내용이 자주 바뀌는 서비스: 블로그, 뉴스 사이트, 쇼핑몰 상품 페이지
    * 예: Next.js SSR 기능, Nuxt.js  
</details>

<details>
  <summary>3. SSG(Static Site Generation)</summary>

  * **빌드 시점에 HTML을 미리 만들어 두고, 요청 시 정적 파일을 전달함**
  * **동작 방식**
    * 빌드할 때 HTML 생성
    * CDN에 저장
    * 요청 시 바로 HTML 반환
  * **특징**
    * 요청 시 서버 연산 없음
    * 정적인 HTML 제공
  * **장점**
    * 매우 빠름(CDN에서 바로 응답)
    * 서버 비용 낮음
    * SEO 매우 좋음
  * **단점**
    * 실시간 데이터 반영 어려움
    * 빌드 시간이 길어질 수 있음
    * 동적 페이지에 부적합
    * 즉, 데이터가 변경되어도 빌드를 다시 하지 않으면 갱신되지 않음
  * **언제 쓰면 좋을까?**
    * 내용이 자주 바뀌지 않는 경우: 기술 블로그, 회사 소개 페이지, 문서 사이트
    * 예: Next.js(getStaticProps), Gatsby 
</details>

<details>
  <summary>4. 핵심 비교</summary>

  |구분|CSR|SSR|SSG|
  |---|---|---|---|
  |렌더링 위치|클라이언트(브라우저)|서버(실시간)|빌드 시(미리)|
  |초기 로딩|느림|빠름|매우 빠름|
  |SEO|나쁨|좋음|매우 좋음|
  |서버 부담|낮음|높음|거의 없음|
  |데이터 최신성|최신|최신|느림|
</details>

<details>
  <summary>5. 선택 기준(실무 기준)</summary>

  * **CSR**: 로그인 기반 서비스/인터랙션 많음, "앱 같은 UX" 필요할 때
  * **SSR**: SEO + 초기 로딩 중요, "검색 노출 + 빠른 첫 화면"
  * **SSG**: 정적 콘텐츠, "속도 + 비용 최적화"
</details>

<details>
  <summary>6. 요즘 트렌드</summary>

  * 하나만 쓰지 않고 **혼합하여 사용함**
  * 예: 메인페이지(SSG) + 상품 페이지(SSR) + 대시보드(CSR)
</details>

<details>
  <summary>7. 정리</summary>

  * CSR/SSR/SSG는 **렌더링 시점과 위치의 차이이고, 상황에 맞게 섞어 쓰는 게 정답임**
</details>
